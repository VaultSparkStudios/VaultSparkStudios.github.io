# VaultSpark Existing Project Migration

Use this process when a project already exists but does not yet follow the full
studio memory system.

## Principle

Backfill from reality, not wishful memory.

Do not attempt perfect archaeology.

## Migration order

1. Add baseline folders if they do not exist
2. Write `PROJECT_BRIEF.md` from current project purpose
3. Write `CURRENT_STATE.md` from actual repo and shipped behavior
4. Create `TASK_BOARD.md` from the real next actions
5. Seed `DECISIONS.md` with only major decisions that are still recoverable
6. Start `LATEST_HANDOFF.md` now
7. Start additive history now in `WORK_LOG.md`

## Non-breaking rule

This migration should be additive first.

Do not reorganize code, rename shipped paths, or move existing public assets
unless there is a specific project decision to do so.
